## Key Design Decisions

### Why Two Python Files?
- **Separation of concerns**: CLI vs Web interface
- **Optional dependencies**: Web server not required for CLI
- **Easier testing**: Can test core logic independently
- **Clearer codebase**: Each file has one clear purpose

### Why JSON Input?
- **Universal format**: Easy to generate from any scanner
- **Human readable**: Can hand-craft test cases
- **Language agnostic**: Works with any scanning tool
- **Simple parsing**: No complex XML or custom formats

### Why Minimal Dependencies?
- **Easy installation**: Fewer things to break
- **Fast setup**: Quick pip install
- **Better security**: Smaller attack surface
- **Wider compatibility**: Works on more systems

## For Contributors

### Adding New Features

1. **New vulnerability types**: Edit `attack_path_mapper.py` line ~78
2. **New ATT&CK techniques**: Edit `AttackTechnique` enum
3. **New visualization**: Edit `web_visualizer.py` HTML section

### Code Style

- **PEP 8 compliant**: Follow Python style guide
- **Type hints**: Use where appropriate
- **Docstrings**: Document all classes and methods
- **Comments**: Explain "why", not "what"
